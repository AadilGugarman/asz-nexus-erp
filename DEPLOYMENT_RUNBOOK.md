# ASZ Nexus ERP Deployment Runbook

Last Updated: May 21, 2026
Owner: Engineering and Release Operations
Scope: Production deployment and recovery for desktop distribution

---

## 1. Purpose

This runbook defines the standard deployment process for ASZ Nexus ERP, including build verification, packaging, release, rollback, and incident response.

Use this document for every production release.

---

## 2. Release Inputs

Required artifacts:

- Signed release installers (MSI and/or NSIS)
- Release notes and changelog
- Security and production readiness validation logs
- Backup and restore validation evidence

Required source references:

- ARCHITECTURE_PRODUCTION_REFERENCE.md
- PRODUCTION_READINESS_SUMMARY.md
- SECURITY_AUDIT_SUMMARY.md
- RELEASE_CHECKLIST.md

---

## 3. Environment Requirements

### 3.1 Build Environment

- Node.js version compatible with package-lock
- Rust toolchain compatible with src-tauri/Cargo.toml
- Windows build agent (for MSI/NSIS packaging)
- Access to signing certificates (if signing is enabled)

### 3.2 Runtime Environment

- Supported Windows version as per product policy
- Disk space for app and local database growth
- Encrypted disk policy recommended for production endpoints

### 3.3 Configuration Requirements

- Production environment variables reviewed
- External API endpoint must be HTTPS only
- Backup destination configured and writable

---

## 4. Pre-Deployment Gate

All items must pass before release:

1. Type check passes.
2. Test suite passes.
3. Frontend production build passes.
4. Rust checks pass.
5. Security audits pass or are explicitly risk-accepted.
6. Backup and restore dry run passes.

Execution commands:

```powershell
npm run typecheck
npm run test
npm run build:prod
cargo check --manifest-path .\src-tauri\Cargo.toml
npm audit
cargo audit
```

If any command fails, stop release and open a blocking issue.

---

## 5. Build and Package Procedure

### 5.1 Clean Build

```powershell
npm ci
npm run build:prod
npm run tauri:build
```

Expected outcome:

- Frontend build artifacts in dist
- Desktop installers generated by Tauri bundler

### 5.2 Validate Package Contents

Confirm:

- App launches successfully
- DB initializes and can read/write data
- Tray icon and window controls function
- Authentication flow works
- Core modules open without crash

---

## 6. Staging Verification

Install on staging machine and execute smoke test:

1. First-time setup and login.
2. Create supplier, customer, and fruit records.
3. Create arrival, purchase invoice, sales invoice.
4. Post payment and verify balances.
5. Run report views.
6. Create backup.
7. Restore backup and verify data integrity.

Record results in release notes.

---

## 7. Production Release Procedure

1. Tag release in version control.
2. Publish installers to distribution channel.
3. Publish release notes.
4. Notify support and operations teams.
5. Monitor first 24h for incidents.

Release metadata to publish:

- Version
- Build timestamp
- Artifact checksums
- Known limitations and mitigations

---

## 8. Post-Release Monitoring

During first 24 hours, monitor:

- Crash frequency
- Authentication failures
- Backup/restore errors
- Database lock or corruption symptoms
- Severe performance regressions

Escalation triggers:

- Data loss risk
- Login failures for valid users
- Repeated startup crash
- Backup restore corruption

---

## 9. Rollback Procedure

Rollback conditions:

- Critical security issue
- Data integrity risk
- Startup failure at scale

Rollback steps:

1. Stop distribution of current release.
2. Re-publish previous stable installer.
3. Notify users and support team.
4. Restore impacted systems from verified backups as needed.
5. Open incident report and root-cause analysis.

Data safety rule:

- Always create an emergency backup before restore or downgrade operation.

---

## 10. Incident Response Workflow

Severity model:

- Sev 1: data corruption, auth bypass, widespread crash
- Sev 2: major feature unavailable, restore failures
- Sev 3: minor regressions with workaround

Response expectations:

- Sev 1: immediate action, same-day mitigation
- Sev 2: fix or workaround within 24h
- Sev 3: schedule in next patch cycle

Incident template:

- Summary
- Impact window
- Affected versions
- Immediate mitigation
- Root cause
- Permanent fix
- Preventive action

---

## 11. Backup and Recovery Policy

Minimum policy:

- Automated daily backup
- Manual backup before upgrades
- 14 to 30 backup retention
- Quarterly restore drills

Recovery objectives (recommended):

- RPO <= 24h
- RTO <= 2h

---

## 12. Compliance and Security Notes

Before each release, confirm:

- CSP remains restrictive
- No debug logs leak sensitive info in production
- Source maps are not shipped
- Token handling policy is documented
- File system access scope is unchanged or reviewed
